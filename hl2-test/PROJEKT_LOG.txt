ZIEL: Half-Life 2 (Source Engine) im Browser lauffähig machen, kompiliert mit
Emscripten zu WebAssembly, mit WebGL2-Rendering und geplanter WebXR-Unterstützung
für Meta Quest 3.

- Referenz: https://github.com/vittorioromeo/HL2VRU (Phase 3)

ASSET-QUELLEN:
- HL2 Retail Release Build 2153 (Archive.org)
  → Direkt extrahierbare Verzeichnisstruktur, kein HLExtract/Wine nötig
  → GCF-Keys: base source shared, materials, models, sounds, hl2 base content

REPOSITORY: https://github.com/Mxthy/hl2-webxr

TECHNISCHE PARAMETER:
- emsdk 3.1.72 (prebuilt, CI)
- PROXY_TO_PTHREAD + SharedMemory (nicht ASYNCIFY)
- INITIAL_MEMORY=2047MB, MAXIMUM_MEMORY=3GB

BLOCKER 1 (KRITISCH): Canvas-Rendering nicht sichtbar
  ─────────────────────────────────────────────────
  Die Engine rendert auf einem Fallback OffscreenCanvas im Worker-Thread,
  der nicht mit der Seite verbunden ist. Der sichtbare <canvas> auf der
  HTML-Seite bleibt leer.

  Versuche:
  1. transferredCanvasNames = "canvas" → pthread_create error 63 → exit
  2. preRun: canvas.transferControlToOffscreen() → Erfolg, aber Worker
     kann nicht auf GL.offscreenCanvases des Main-Threads zugreifen
  3. transferredCanvasNames filter + preRun transfer → gleicher Fehler
  4. Module.canvas = null → Engine findet Canvas nicht
  5. Dummy canvas als Module.canvas → gleicher Transfer-Fehler
  6. webglcontextlost listener entfernt → keine Besserung

  Kernproblem: Etwas nimmt Kontrolle über den Canvas, bevor pthread_create
  läuft. transferControlToOffscreen schlägt fehl mit "current thread does

LÖSUNG ZU BLOCKER 1 (Canvas-Rendering):
  Option A: OffscreenCanvas Blit
    - Engine rendert auf Fallback OffscreenCanvas im Worker
    - Worker nutzt OffscreenCanvas.transferToImageBitmap()
    - postMessage(ImageBitmap) an Main-Thread
    - Main-Thread zeichnet auf sichtbaren Canvas via drawImage()
    - Pro: Funktioniert ohne Canvas-Transfer
    - Contra: Performance-Overhead, benötigt Render-Loop-Hook

  Option B: Module.canvas als OffscreenCanvas
    - Main-Thread erstellt OffscreenCanvas via new OffscreenCanvas(1280,800)
    - Setzt Module.canvas = offscreenCanvas (nicht HTMLCanvas)
    - Worker erhält OffscreenCanvas via structured clone in pthread_create
    - Main-Thread nutzt bitmaprenderer context zum Anzeigen
    - Pro: Direkter Rendering-Pfad
    - Contra: OffscreenCanvas kann nicht via transferControlToOffscreen
      vom HTML-Canvas erstellt werden

  Option C: Manual Canvas Transfer
    - Vor pthread_create: canvas.transferControlToOffscreen()
    - OffscreenCanvas in globaler Variable speichern
    - Worker postMessage mit OffscreenCanvas im transfer list
    - findCanvasEventTarget im Worker gibt transferierten Canvas zurück
    - Pro: Korrekter Emscripten-Weg
    - Contra: Muss pthread_create patchen

  Option D: Emscripten _emscripten_set_offscreencanvas_size()
    - Engine ruft _emscripten_set_canvas_size_on_main_thread() auf
    - Patch umleiten auf Fallback OffscreenCanvas Größe
    - Dann Blit auf sichtbaren Canvas
    - Pro: Minimal-invasiv
    - Contra: Zeigt nur Größe, nicht Inhalt

  EMPFOHLEN: Option A (OffscreenCanvas Blit) als quickest win,
  dann Option B oder C für proper Lösung.

LÖSUNG ZU BLOCKER 2 (Chunk-Loading):
  1. R2-Streaming mit Range-Requests testen
  2. Chunk-Größe reduzieren (background1 in 2x400MB splitten)
  3. OPFS (Origin Private File System) Caching für wiederholte Loads
  4. Fetch-Timeout erhöhen (aktuell 30s für background1)
  5. Memory-Limit des Headless-Browsers erhöhen
  6. Bei echtem Browser testen (nicht Headless)
